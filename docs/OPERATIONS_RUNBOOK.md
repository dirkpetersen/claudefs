## Daily Operations Checklist

### Morning (Start of Shift)

**Time:** 09:00 UTC (or shift start)
**Duration:** 10 minutes
**Owner:** On-call operator

#### 1. Cluster Health Overview

```bash
ssh orchestrator

# Quick cluster status
cfs-dev status

# Expected output:
# Orchestrator: running ✓
# Nodes: 8/8 healthy ✓
# Replication: all sites in sync ✓
# Storage: 60% used (normal range: 40-80%)
```

#### 2. Check Monitoring Alerts

```bash
# View all active alerts
curl -s http://prometheus:9090/api/v1/alerts | jq '.data.alerts'

# Expected: 0 alerts (or only info-level)
# If CRITICAL alert: page incident lead immediately
```

#### 3. Review Overnight Logs

```bash
# Check for errors in overnight logs
ssh orchestrator
grep -i "error\|panic\|crash" /var/log/cfs-agents/*.log | tail -20

# Expected: no ERROR or PANIC messages
# If found: investigate using DEBUGGING_RUNBOOK.md
```

#### 4. Verify Replication Status

```bash
# Check cross-site replication
cfs admin show-replication-status

# Expected output:
# Site A (primary): 3 nodes, 1 leader
# Site B (backup): 2 nodes, online
# Replication lag: 0-500ms (target <1s)

# If lag >5s: trigger alert, see DEBUGGING_RUNBOOK.md
```

#### 5. Quick Performance Baseline

```bash
# Spot-check latency (p99)
curl -s 'http://prometheus:9090/api/v1/query?query=histogram_quantile(0.99,storage_latency_seconds)' | jq '.data.result[].value'

# Expected: <1000ms (milliseconds)
# If >2s: performance degraded, investigate node metrics
```

### Midday (Standby)

**Time:** 12:00-18:00 UTC
**Duration:** On-demand
**Tasks:**

- Monitor metrics dashboard (refresh every hour)
- Respond to alerts immediately
- Check email for incidents/escalations
- Run weekly maintenance tasks (if applicable)

### Evening (End of Shift)

**Time:** 18:00 UTC (before shift end)
**Duration:** 15 minutes
**Owner:** On-call operator

#### 1. Handoff to Next Shift

```bash
# Summary of shift
echo "
=== Shift Summary ===
Date: $(date)
Incidents: 0
Alerts triggered: 0
Performance: normal
Replication: in-sync

=== Next Shift Checklist ===
- [ ] Run morning cluster health check
- [ ] Review overnight logs (will be generated by then)
- [ ] Verify replication

=== Known Issues ===
None
" > /tmp/shift-handoff.txt

# Share with next on-call
# post to #operations Slack
```

#### 2. Archive Logs

```bash
# Compress logs from past 24 hours
tar czf /tmp/logs-$(date +%Y-%m-%d).tar.gz /var/log/cfs-agents/

# Upload to S3 (if needed)
aws s3 cp /tmp/logs-$(date +%Y-%m-%d).tar.gz s3://claudefs-ops-logs/
```

## Weekly Maintenance

### Every Monday (Weekly Health Check)

**Duration:** 1 hour
**Owner:** SRE team

#### 1. Verify Backup Integrity

```bash
# Full backup check
cfs-dev backup --full --verify

# Expected:
# Total backup size: <500GB
# Verification: PASSED
# Recovery time estimate: <30min

# Expected output:
# Backup OK
# Redundancy: 4+2 EC (2 stripes on S3)
```

#### 2. Run Disaster Recovery Drill

```bash
# Restore to staging environment
cfs-dev provision-staging
cfs-dev restore-from-backup --env staging --date latest

# Verify recovery
cfs-dev test --env staging --quick

# Expected:
# Data restored: ✓
# Tests passing: ✓
# Recovery time: <30 min

# Cleanup
cfs-dev destroy-staging
```

#### 3. Security Audit

```bash
# Check for CVEs in dependencies
cargo audit

# Expected: No CVE entries

# Check security logs
grep -i "authentication\|authorization\|denied" /var/log/cfs-agents/*.log | tail -20

# Expected: No authentication failures
```

#### 4. Capacity Planning Review

```bash
# Query storage usage trend over past week
prometheus_query() {
  curl -s "http://prometheus:9090/api/v1/query_range?query=$1&start=$(date -d '7 days ago' +%s)&end=$(date +%s)&step=3600" | jq
}

prometheus_query 'storage_capacity_used_bytes'

# Expected: stable or slow growth
# If >5% growth per day: schedule scaling
```

#### 5. Performance Regression Check

```bash
# Compare latencies to baseline (from METRICS-INTEGRATION-GUIDE.md)
# Baseline: p99 latency <1s, throughput >1000 ops/sec

prometheus_query 'histogram_quantile(0.99, storage_latency_seconds)'
prometheus_query 'rate(operations_total[1m])'

# Expected: within 10% of baseline
# If >10% worse: investigate (see DEBUGGING_RUNBOOK.md)
```

## Scaling Operations

### Add a New Storage Node

**When:** Storage >80% full, or latency >2s
**Time:** ~30 minutes
**Owner:** SRE + A11

**Step 1: Provision new node**

```bash
cfs-dev scale --nodes 9  # Add 1 node (8 → 9)

# Verify instance provisioned
aws ec2 describe-instances --filters Name=tag:role,Values=storage | jq '.Reservations[].Instances[].InstanceId'

# Expected: new i-xxxxx ID shown
```

**Step 2: Verify cluster integration**

```bash
# Wait for SWIM to discover new node
sleep 30

# Check cluster status
cfs-dev status

# Expected:
# Nodes: 9/9 healthy
# Rebalancing: in-progress

# Monitor rebalancing (should take 5-15 min)
watch 'cfs admin show-rebalancing-status'
```

**Step 3: Validate performance restored**

```bash
# Check latency after rebalancing complete
curl -s 'http://prometheus:9090/api/v1/query?query=histogram_quantile(0.99,storage_latency_seconds)' | jq '.data.result[].value'

# Expected: back to <1000ms

# Check storage now balanced
cfs admin show-node-storage | awk '{print $1, $2}'

# Expected: all nodes have roughly equal usage (±5%)
```

### Remove a Node (Graceful Decommission)

**When:** Scheduled maintenance, or node failing
**Time:** ~15-20 minutes
**Owner:** SRE

**Step 1: Mark node for removal**

```bash
# Graceful drain (migrate data off node)
cfs admin drain-node storage-node-5

# Monitor drain progress
watch 'cfs admin show-drain-status storage-node-5'

# Expected: 100% after 10-15 min
```

**Step 2: Remove from cluster**

```bash
# Once drain complete
cfs admin remove-node storage-node-5

# Verify
cfs-dev status

# Expected:
# Nodes: 8/8 healthy (1 removed)
```

**Step 3: Terminate instance**

```bash
# Get instance ID
aws ec2 describe-instances --filters Name=tag:Name,Values=storage-node-5 \
  --query 'Reservations[].Instances[].InstanceId' | jq -r '.[0]'

# Terminate
aws ec2 terminate-instances --instance-ids i-xxxxx

# Verify
aws ec2 describe-instances --instance-ids i-xxxxx | jq '.Reservations[].Instances[].State.Name'

# Expected: terminated
```

## Monitoring & Alerting Setup

### Prometheus Alert Rules

**File:** `monitoring/prometheus/rules.yml`

```yaml
groups:
  - name: claudefs-production
    interval: 30s
    rules:
      # Rule: High latency
      - alert: P99LatencyHigh
        expr: histogram_quantile(0.99, rate(storage_latency_seconds[5m])) > 1.0
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "P99 latency > 1 second"

      # Rule: Replication lag
      - alert: ReplicationLagHigh
        expr: max(replication_lag_seconds) > 5
        for: 2m
        labels:
          severity: critical
        annotations:
          summary: "Cross-site replication lag > 5s"

      # Rule: Disk nearly full
      - alert: DiskSpaceWarning
        expr: (storage_capacity_total - storage_capacity_used) / storage_capacity_total < 0.2
        for: 10m
        labels:
          severity: warning
        annotations:
          summary: "Disk space < 20% remaining"
```

### Alert Routing (Alertmanager)

**File:** `monitoring/alertmanager/config.yml`

```yaml
global:
  slack_api_url: 'https://hooks.slack.com/services/YOUR/WEBHOOK/URL'

route:
  receiver: 'critical-ops'
  group_by: ['alertname', 'cluster']
  routes:
    - match:
        severity: critical
      receiver: 'ops-pagerduty'
      continue: true

    - match:
        severity: warning
      receiver: 'ops-slack'

receivers:
  - name: 'critical-ops'
    pagerduty_configs:
      - service_key: 'YOUR_SERVICE_KEY'

  - name: 'ops-pagerduty'
    pagerduty_configs:
      - service_key: 'YOUR_SERVICE_KEY'
    slack_configs:
      - channel: '#ops-critical'

  - name: 'ops-slack'
    slack_configs:
      - channel: '#ops-warnings'
```
